# Simulator Architecture

Plan: `GLP-0003`

## Purpose

The simulator has two jobs:

1. Prove rust-orbitdb correctness under controlled fault schedules before any
   real network is involved.
2. Let humans and agents compare adaptation strategies for throughput, latency,
   churn, reattach behavior, and failure recovery without requiring a physical
   million-peer network.

The simulator MUST be able to run tiny cases (`N=2`, `N=3`) and large cases
using the same scenario definitions. High-scale failures SHOULD become small
unit or deterministic regression tests.

## Variant A: Direct Semantic Simulator

`rust-orbitdb-sim` implements the rust-orbitdb substrate trait directly. It does
not instantiate libp2p and does not simulate every byte of a network stack.

Primary use:

- exact schedule tests
- append/join interleaving
- storage crash boundaries
- data availability and last-holder-departs faults
- delivery drop, duplicate, reorder, delay
- partition/heal
- random start/stop/restart
- session lifecycle and reattach
- throughput/latency modeling for strategy comparison
- compressed Monte Carlo to million-scale

Expected strengths:

- deterministic and reproducible
- small enough to reason about
- fast enough for CI and high-scale sweeps
- able to shrink high-scale failures into small traces

Expected limits:

- may miss libp2p-specific stream, swarm, muxer, peer discovery, and backpressure
  behavior
- can encode the same wrong assumptions as the implementation if not checked
  against JS OrbitDB and libp2p-facing tests

## Variant B: In-Memory libp2p Semantics Simulator

`rust-orbitdb-libp2p-sim` SHOULD exercise libp2p-facing adapter logic over
in-memory links/failures without real TCP/UDP sockets. The local rust-libp2p
source already includes `MemoryTransport` in
`third-party/rust-libp2p/core/src/transport/memory.rs`, and swarm tests use it
as an in-memory transport. This is the first implementation path to investigate.

Primary use:

- adapter integration depth without real sockets
- swarm/protocol behavior under simulated link failures
- adapter-specific protocol, stream, notification, and peer lifecycle pressure
- testing whether libp2p instance overhead permits useful in-memory scale

Expected strengths:

- exercises more of the real libp2p stack than the direct simulator
- can reveal adapter assumptions that the direct simulator cannot
- avoids OS networking limits for many integration tests

Expected limits:

- likely cannot reach the same scale as compressed direct simulation
- may still carry substantial per-instance memory and scheduler overhead
- may require custom link/failure injection around existing memory transport

This variant SHOULD be treated as an integration stack simulator, not the only
scale simulator.

## Node.js JS OrbitDB Runner

`node/rust-orbitdb-sim-node` MUST run selected JS OrbitDB scenarios through the
same scenario definitions used by Rust tests. This is mainly to test the test
harness.

Required behavior:

- load scenario definitions generated by `rust-orbitdb-testkit`
- instantiate JS OrbitDB without modifying vendored source
- drive operations and simulated delivery through a Node-facing adapter
- emit comparable traces: operations, heads, CIDs, causal edges, convergence
  state, timing buckets, and failures
- fail if a deliberate simulator mutation hides a known JS OrbitDB behavior

The Node runner MAY use JS OrbitDB and js-libp2p dependencies in the Node
package. It MUST NOT force Rust fixture/core crates to depend on libp2p.

## Shared Scenario DSL

All runners SHOULD consume one scenario model:

```text
scenario
  nodes
  sessions/logs/documents
  operations
  delivery schedule
  failures
  assertions
  metrics
```

The scenario model MUST support:

- exact schedule replay
- pseudo-random generation with seed
- fault injection
- metric collection
- trace comparison between Rust, JS, direct sim, libp2p sim, and real libp2p
- minimizing or extracting small regressions from high-scale failures

The scenario DSL MUST be versioned before simulator implementation. It MUST
define:

- schema version
- supported operation families
- deterministic seed format
- participant/session/document identifiers
- delivery/failure schedule format
- assertion format
- metric names and units
- trace schema version

## Trace Comparison

Rust and JS traces MUST be compared by a normative schema. Comparable fields
include:

- scenario id and version
- seed
- operation id
- participant id
- logical session/log/document id
- entry CID or document operation id when available
- causal parent ids
- observed heads
- document visible state
- failure class
- convergence status

Ignored or bucketed fields MUST be explicit. Wall-clock timings SHOULD be
reported as metric buckets, not exact equality. Implementation-specific object
ids, process ids, memory addresses, and non-deterministic debug strings MUST NOT
participate in equality.

If Rust and JS disagree, the harness MUST classify the disagreement before the
checkpoint passes:

- expected compatibility delta recorded in `Decisions.md`
- simulator bug
- Rust implementation bug
- JS runner limitation
- trace-schema limitation

## Exact-To-Compressed Bridge

Compressed simulation MUST preserve the invariant set from exact small-N runs.
Before a compressed million-scale run can be accepted, the same scenario family
MUST pass:

- exact `N=2`
- exact `N=3`
- exact sampled session inside a larger compressed run
- compressed high-N run with aggregate counters

The bridge MUST assert that compression does not hide:

- invalid entry acceptance
- lost head updates
- document convergence failures
- unauthorized operations
- missing content-addressed block fetch or silent data-unavailability handling
- event queue blowups
- latency or memory threshold violations

Every accepted compressed result MUST report the exact invariant set it claims to
preserve and the sampled exact sessions used to check that claim.

## Data Availability Faults

Convergence assertions MUST name their data-availability assumption. The default
modeled assumption is that every entry/block referenced by a delivered head
remains retrievable from at least one reachable peer or storage tier within the
modeled fetch window.

The simulator MUST also include negative faults that violate that assumption,
including a last holder departing before other peers fetch the referenced block
and a storage tier refusing or losing a referenced block. Those runs MUST assert
typed unavailability and bounded retry behavior, not silent divergence.

## Adaptation Strategy Experiments

The simulator SHOULD support product-agnostic adaptation scenarios shaped like:

- open session
- live input stream
- output append-log
- replay from cursor
- live-tail cutover
- resize/control signal
- stop/restart
- reattach
- multi-participant read fan-out
- single-driver or arbitration variants

These scenarios MUST avoid downstream product naming. They exist to test
strategy choices, not to encode one product contract.

## Metrics

Every simulator variant SHOULD report:

- delivered messages
- dropped/duplicated/reordered messages
- content-fetch latency
- unavailable block count
- live-stream latency
- replay duration
- time to convergence
- memory estimate
- active participants per session
- inactive sessions/clients represented by compressed state
- event queue depth
- backpressure events
- restart and reattach success/failure counts

## Acceptance

The simulator architecture is accepted when:

- the same scenario can run at `N=2`, `N=3`, and larger configured `N`
- exact and compressed runs share a named invariant set
- a high-N sweep records the scaling boundary and the limiting bottleneck, and
  named hot paths are shown sub-quadratic with live state O(active) (see D0015);
  no fixed peer count is required to pass
- direct sim can run without libp2p
- Node runner can execute selected JS OrbitDB scenarios
- in-memory libp2p simulator can run adapter-facing scenarios without real
  sockets, or the plan records why existing libp2p memory transport cannot meet
  the requirement
- high-scale failure traces can be replayed and reduced into small regressions
