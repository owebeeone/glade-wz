# Effect Request & Remote Tap RPC Protocol (ERP)
**Document ID:** GLIAL-DOC-006
**Version:** 1.0
**Status:** Contract Specification
**Dependency:** GLIAL-DOC-001, GLIAL-DOC-002, GLIAL-DOC-003, GLIAL-DOC-005

---

## 1. Introduction

The **Effect Request Protocol (ERP)** defines the contract between the **State Plane** (Engine) and the **Effect Plane** (Provider). It also defines RPC-style invocation of remote taps for imperative operations.

ERP has two request families:
* **State-driven Effect Request:** Engine -> Provider (`EFFECT_REQ`) for dependency resolution.
* **Caller-driven Remote Tap RPC:** Caller -> Provider via Relay (`RPC_REQ`) for imperative commands.

---

## 2. Shared Envelope Fields

All ERP requests MUST include:
* `req_id` or `rpc_id` (correlation id).
* `tap` or `grip` target.
* `context.trace_id` for distributed tracing (when available).
* `timeout_ms` (explicit or defaulted by Relay/Engine policy).

---

## 3. State-Driven Effect Request (`EFFECT_REQ`)

When the Engine determines a Grip needs data (and has dependencies satisfied), it sends:

```json
["EFFECT_REQ", {
  "req_id": "eff_999",
  "grip": "data.user.profile",
  "params": { "user_id": 123 },
  "context": { "trace_id": "t_abc" }
}]
```

---

## 4. Effect Response Contract

The Provider MUST respond with one of the following `OP` types targeting the requested Grip.

### 4.1 Success
```json
["OP", { "verb": "SET", "grip": "data.user.profile", "val": { ... }, "ref_req": "eff_999" }]
```
* **Note:** The `ref_req` field ties the result to the specific request, allowing the Engine to measure latency and clear pending states.

### 4.2 Failure (Retryable)
```json
["ERR", { "code": "E_TIMEOUT", "retry_after": 500, "ref_req": "eff_999" }]
```
* The Engine will schedule a retry.

### 4.3 Failure (Terminal)
```json
["ERR", { "code": "E_NOT_FOUND", "msg": "User 123 does not exist", "ref_req": "eff_999" }]
```
* The Engine sets the Grip state to `error` and propagates this to the UI.

---

## 5. Remote Tap RPC Contract (`RPC_REQ` / `RPC_RES`)

### 5.1 Request
Caller (browser, backend client, or agent) sends:
```json
["RPC_REQ", {
  "rpc_id": "rpc_01HZX2...",
  "tap": "rpc.user.create",
  "args": { "email": "a@example.com" },
  "timeout_ms": 5000,
  "idempotency_key": "idem_2e0d..."
}]
```

Relay MUST:
* Validate caller capability/scope for this invocation (Doc 3).
* Route to an eligible Provider (Doc 5).
* Preserve `rpc_id` for end-to-end correlation.

### 5.2 Success
```json
["RPC_RES", {
  "rpc_id": "rpc_01HZX2...",
  "ok": true,
  "result": { "user_id": "u_123" }
}]
```

### 5.3 Application Failure
```json
["RPC_RES", {
  "rpc_id": "rpc_01HZX2...",
  "ok": false,
  "error": { "code": "E_VALIDATION", "msg": "email already used" },
  "retryable": false
}]
```

### 5.4 Protocol/Security Failure
```json
["ERR", { "code": "E_ACCESS_DENIED", "ref_rpc": "rpc_01HZX2..." }]
```

### 5.5 Timeout and Cancellation
* If timeout elapses without terminal response, Relay MUST return:
```json
["ERR", { "code": "E_RPC_TIMEOUT", "ref_rpc": "rpc_01HZX2..." }]
```
* Caller MAY send `RPC_CANCEL` for in-flight calls.
* Provider SHOULD treat cancellation as best-effort; non-cancellable work MAY still complete.

### 5.6 State Interaction Rule
`RPC_RES(ok=true)` acknowledges command completion, not automatic graph mutation.
If the RPC changes authoritative state, Provider MUST emit explicit `OP` writes.

---

## 6. Standard Error Codes

* `E_ACCESS_DENIED`: Caller lacks required capability/scope.
* `E_NOT_PROVIDED`: No Provider currently registered for target tap/grip.
* `E_RPC_TIMEOUT`: RPC timed out before terminal response.
* `E_VALIDATION`: Request arguments invalid.
* `E_CONFLICT`: Concurrency or idempotency conflict.
* `E_PROVIDER_UNAVAILABLE`: Provider disconnected or lease lost during execution.

---

## 7. Provenance & Audit Metadata
For AI Agents, it is critical to know *where* data came from.
Providers SHOULD include a `provenance` metadata field in `OP` and/or `RPC_RES`:
```json
"meta": {
  "source": "postgres_primary",
  "latency_ms": 45,
  "generated_by": "model:gpt-4o" // If generated by AI
}
```

Relay/Engine MUST be able to audit:
* caller identity and capabilities,
* target tap/grip,
* correlation id (`req_id`/`rpc_id`),
* terminal outcome code,
* timestamp.
