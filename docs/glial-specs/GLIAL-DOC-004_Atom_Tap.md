# AtomTap Operation Semantics
**Document ID:** GLIAL-DOC-004

**Status:** Contract Specification

## 1. Introduction

While GSP defines *how* to send an operation, this document defines *what* the operations mean. Specifically, it defines the semantics of **AtomTaps**—shared, mutable state primitives.

---

## 2. Operation Types

The `OP` payload in GSP contains a `verb`. Glial supports a limited set of verbs to ensure determinism.

### 2.1 Primitive Verbs

| Verb | Payload | Semantics | Conflict Resolution |
| :--- | :--- | :--- | :--- |
| `SET` | `{ val: any }` | Overwrites the value. | **Last-Write-Wins (LWW)** based on Server Arrival Time. |
| `DEL` | `null` | Removes the node. | LWW. |
| `MERGE`| `{ val: obj }` | Shallow merge into an object. | LWW per key. |

### 2.2 Atomic Verbs (The "CAS" Suite)

| Verb | Payload | Semantics | Conflict Resolution |
| :--- | :--- | :--- | :--- |
| `CAS` | `{ old: v, new: v }` | Compare-And-Swap. | **Strict.** Succeeds only if current val == `old`. |
| `INC` | `{ delta: num }` | Numeric increment. | **Commutative.** Always succeeds. |
| `PUSH` | `{ item: any }` | Append to list. | **Commutative.** Order determined by arrival. |

## 3. Idempotency & Ordering

### 3.1 Op ID
Every `OP` message generated by a Client includes a temporary `req_id`.
When the Server processes it, it assigns a permanent `op_id`.
The Server broadcasts the result back with *both* IDs: `["OP", { "op_id": 105, "req_id": "abc-123", "verb": "SET", ... }]`.

### 3.2 Replayability
The State Plane must apply ops in `op_id` order.
* **AtomTaps:** Are strictly ordered.
* **ProviderTaps:** Writes from providers are authoritative and generally effectively `SET` operations that supersede previous state.

---

## 4. Destination Parameters (Routing Ops)

Some Ops are not meant to change a value, but to **change a destination**.
* **Scenario:** A `DestinationTap` (e.g., Fetch User) depends on `user_id`.
* **Mechanism:** When `user_id` changes, the Engine treats this change as a **Re-Route Op**.
* **Effect:** It triggers a new `Effect Request` (see Doc 6) with the new parameter.
